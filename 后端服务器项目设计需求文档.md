# Linux C++ 后台服务器项目设计需求文档（V1.0）

## 1. 项目目标
构建一个可在 **WSL2 / Ubuntu** 运行的高性能后台服务，支持 **Web** 与 **Android** 客户端通过 **HTTPS** 请求访问，首期实现“手机号 + 密码”的注册与登录能力，并具备数据库管理与后续扩展基础。

## 2. 范围与阶段
### 2.1 首期（MVP）
- 用户注册（手机号 + 密码）
- 用户登录（手机号 + 密码）
- JWT 鉴权与基础权限控制
- 用户信息查询（用于验证登录态）
- HTTPS 服务部署
- MySQL 数据库管理（建表、索引、迁移脚本）
- 基础日志、配置、健康检查接口

### 2.2 二期（可扩展）
- 短信验证码注册/登录
- 密码找回
- 登录风控（限流、黑名单、设备指纹）
- 审计日志、监控告警、灰度发布

## 3. 技术要求
### 3.1 开发语言与标准
- 语言：`C++17`
- 构建：`CMake >= 3.20`
- 编译器：`g++ >= 11` 

### 3.2 服务框架与核心组件（建议）
- 网络层：Boost.Asio（异步 IO，多路复用，epoll）
- HTTP/HTTPS：Boost.Beast 或 Drogon（二选一，建议 Drogon 提升开发效率）
- TLS：OpenSSL 3.x
- 数据库：MySQL 8.x（可选 MariaDB 10.6+）
- 连接池：自研池或框架池
- 序列化：JSON（nlohmann/json 或框架内置）
- 密码哈希：`Argon2id`（优先）或 `bcrypt`
- Token：JWT（RS256 或 HS256）

### 3.3 系统环境
- Ubuntu 22.04 
- WSL2 Ubuntu 22.04
- 支持 systemd（原生 Ubuntu）与手动进程方式（WSL）

## 4. 功能需求
### 4.1 用户注册
- 输入：手机号、密码、确认密码
- 校验规则：
  - 手机号：11 位中国大陆号码（正则可配置）
  - 密码：8~64 位，至少包含字母和数字
  - 不允许重复手机号注册
- 存储规则：
  - 仅存储密码哈希与盐，不落库存明文密码
- 输出：注册成功/失败及错误码

### 4.2 用户登录
- 输入：手机号、密码
- 校验：
  - 手机号存在
  - 密码哈希比对通过
- 输出：
  - `access_token`（JWT）
  - `expires_in`

### 4.3 用户信息接口
- 需携带 Bearer Token
- 返回基础用户信息（id、手机号、创建时间）

### 4.4 健康检查
- `GET /health`
- 返回服务运行状态、版本、数据库连接状态

## 5. 接口设计（REST）
统一前缀：`/api/v1`

### 5.1 注册
- `POST /api/v1/auth/register`
- Request(JSON):
```json
{
  "phone": "13800138000",
  "password": "Abc12345",
  "confirm_password": "Abc12345"
}
```
- Response(JSON):
```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "user_id": 10001
  }
}
```

### 5.2 登录
- `POST /api/v1/auth/login`
- Request(JSON):
```json
{
  "phone": "13800138000",
  "password": "Abc12345"
}
```
- Response(JSON):
```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "access_token": "xxx.yyy.zzz",
    "token_type": "Bearer",
    "expires_in": 7200
  }
}
```

### 5.3 当前用户信息
- `GET /api/v1/users/me`
- Header: `Authorization: Bearer <token>`

## 6. 数据库设计
### 6.1 表结构：`users`
```sql
CREATE TABLE users (
  id BIGINT UNSIGNED PRIMARY KEY AUTO_INCREMENT,
  phone VARCHAR(20) NOT NULL,
  password_hash VARCHAR(255) NOT NULL,
  password_algo VARCHAR(20) NOT NULL DEFAULT 'argon2id',
  status TINYINT NOT NULL DEFAULT 1,
  created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  UNIQUE KEY uk_phone (phone)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

### 6.2 可选表：`login_logs`
- 记录 user_id、ip、ua、登录结果、时间戳，用于审计与风控。

## 7. 安全要求
- 全链路 HTTPS（禁用明文 HTTP 对外暴露）
- TLS 最低版本：1.2（建议 1.3）
- 密码哈希：Argon2id（必须带盐）
- JWT 签发与校验，支持过期与撤销（黑名单可后续实现）
- 防暴力破解：登录失败次数限制、IP/手机号限流
- 输入校验与参数化查询，防注入
- 敏感日志脱敏（手机号中间位打码）

## 8. 高性能与并发设计
### 8.1 并发模型
- Reactor 模型 + 线程池
- 使用 epoll（Linux）进行多路复用
- 网络 IO 与业务逻辑线程隔离

### 8.2 性能目标（MVP）
- 单机（4C8G）下：
  - 读写混合场景吞吐：`>= 3,000 QPS`
  - P95 延迟：`< 50ms`
  - P99 延迟：`< 120ms`
- 支持连接池、对象复用、零拷贝优化（框架能力优先）

### 8.3 稳定性
- 异常请求不导致进程崩溃
- 关键模块具备超时、重试、熔断（可先实现超时 + 降级）

## 9. 可观测性与运维
- 日志：
  - 访问日志、错误日志、审计日志分离
  - 支持按天切分与保留策略
- 监控：
  - QPS、错误率、延迟分位数、连接池状态、DB 慢查询
- 健康探针：
  - 存活探针 `GET /health/live`
  - 就绪探针 `GET /health/ready`

## 11. 编译与运行要求（WSL/Ubuntu）
### 11.1 依赖安装（示例）
```bash
sudo apt update
sudo apt install -y build-essential cmake ninja-build git pkg-config \
  libssl-dev libboost-all-dev libmysqlclient-dev
```

### 11.2 编译
```bash
cmake -S . -B build -G Ninja -DCMAKE_BUILD_TYPE=Release
cmake --build build -j
```

### 11.3 运行
```bash
./build/server --config ./config/app.yaml
```

### 11.4 WSL 注意事项
- WSL 可直接监听 `0.0.0.0:1818`，Windows 侧通过 `localhost:1818` 访问。
- 证书可先使用自签名用于开发环境，生产环境必须使用正规 CA 证书。

## 12. 配置项需求（示例）
```yaml
server:
  host: 0.0.0.0
  port: 1818
  worker_threads: 8
  tls_cert: ./cert/server.crt

  port: 3306

  dbname: app_db
  pool_size: 20

  jwt_expire_seconds: 7200

```

## 13. 测试与验收标准
### 13.1 测试类型
- 单元测试：手机号校验、密码策略、哈希校验、JWT
- 集成测试：注册/登录全链路 + 数据库交互
- 压力测试：`wrk` / `hey` 对登录接口并发压测
- 安全测试：弱密码、暴力破解、SQL 注入、越权访问

### 13.2 验收标准（MVP）
- 在 Ubuntu 与 WSL 环境均可编译通过并成功启动
- 注册、登录、鉴权接口可用且返回码规范一致
- HTTPS 可用，证书配置正确
- 数据库存储不含明文密码
- 基础性能指标达到第 8 节目标

## 14. 风险与对策
- OpenSSL/数据库版本差异导致编译失败：
  - 对策：固定依赖版本并提供 Docker 开发镜像（可选）
- 高并发下数据库瓶颈：
  - 对策：连接池 + 索引优化 + 缓存预留
- Token 泄露风险：
  - 对策：缩短有效期 + HTTPS + 密钥轮换

## 15. 里程碑建议
- M1（第 1 周）：项目骨架、HTTPS 服务、数据库连通
- M2（第 2 周）：注册/登录接口、JWT、基础测试
- M3（第 3 周）：限流、日志、健康检查、压测优化
- M4（第 4 周）：联调、部署文档、验收交付
